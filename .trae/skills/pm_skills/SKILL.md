# 产品经理专业技能文档

## 技能1：需求诊断与价值评估（核心能力）

### 技能描述
这是20年资深产品经理最核心的能力——在动手之前，先判断需求是否值得做、怎么做、做到什么程度。

### 具体能力

#### 1.1 需求价值评估
- **业务价值判断**：这个需求解决什么问题？用户痛点有多痛？市场规模有多大？
- **投入产出分析**：开发成本是多少？预期收益是多少？优先级如何？
- **竞争壁垒分析**：做这个功能能形成护城河吗？还是只能抄竞争对手？

#### 1.2 需求边界界定
- **MVP定义**：最小可行产品应该包含什么？不包含什么？
- **分期规划**：第一期做什么？第二期做什么？roadmap怎么规划？
- **风险预判**：做这个需求可能有哪些风险？怎么规避？

#### 1.3 干系人分析
- **决策链梳理**：谁决策？谁使用？谁受益？谁反对？
- **利益诉求挖掘**：各方干系人想要什么？怎么平衡各方利益？
- **决策链梳理**：谁决策？谁使用？谁受益？谁反对？
- **利益诉求挖掘**：各方干系人想要什么？怎么平衡各方利益？

### 使用工具
- Kaiwu Form MCP：收集各方需求
- WebSearch MCP：竞品分析、市场调研
- Write：生成需求分析报告

### 工作流程
1. 收集原始需求（用户访谈、反馈、竞品分析）
2. 挖掘真实痛点（5Why分析法）
3. 评估业务价值（成本收益分析）
4. 界定需求边界（MVP定义）
5. 输出需求评估报告

### 输出标准
- 需求评估报告.md
  - 需求背景（谁提的？为什么提？）
  - 用户痛点（真实需求是什么？）
  - 业务价值（能解决什么问题？）
  - MVP范围（第一期做什么？）
  - 优先级建议（P0/P1/P2/P3）
  - 风险提示（可能有什么问题？）

## 技能2：需求收集与分析

### 技能描述
从多个渠道系统性地收集需求，并进行深度分析，区分真需求与伪需求。

### 具体能力

#### 1.1 需求来源收集
- **用户访谈**：一对一深度访谈，理解用户真实场景
- **问卷调查**：大规模收集，量化用户需求
- **数据分析**：从现有数据中发现问题机会点
- **竞品分析**：研究竞争对手，发现差异化机会
- **运营反馈**：收集一线运营人员的实际问题
- **技术洞察**：从技术角度发现体验优化点

#### 1.2 需求分类与优先级
- **需求类型区分**
  - 基础型需求：竞品有我们也必须有
  - 期望型需求：用户期望我们做得更好
  - 兴奋型需求：用户没想到但很惊喜
  - 无差异型需求：做了没好处，不做没坏处

- **Kano模型应用**
  - Must-be：必须有，不满意
  - One-dimensional：越有越满意
  - Attractive：越有越惊喜
  - Indifferent：有没有都无所谓
  - Reverse：反向特征

- **优先级矩阵**
  - 价值高 + 成本低 → 立即做（P0）
  - 价值高 + 成本高 → 规划做（P1）
  - 价值低 + 成本低 → 考虑做（P2）
  - 价值低 + 成本高 → 不做（P3）

#### 1.3 需求文档化
- **用户故事**：As a [角色], I want to [功能], so that [收益]
- **验收标准**：Given-When-Then 格式
- **功能清单**：ID、名称、描述、优先级、验收标准

### 使用工具
- Kaiwu Form MCP：需求收集与管理
- WebSearch MCP：竞品分析与市场调研
- Write：生成需求文档

### 工作流程
1. 确定需求来源（访谈/问卷/数据/竞品）
2. 收集原始需求
3. 分类整理需求
4. 评估优先级
5. 编写需求文档

### 输出标准
- 需求分析报告.md
  - 需求来源（谁提的？从哪来的？）
  - 需求分类（什么类型的需求？）
  - 优先级（为什么是这个优先级？）
  - 详细需求列表（用户故事+验收标准）

## 技能3：业务流程设计

### 技能描述
设计清晰、高效的业务流程，包括正常流程、分支流程和异常流程。

### 具体能力

#### 2.1 流程梳理方法
- **端到端流程**：从用户角度出发，梳理完整业务流程
- **泳道图**：清晰展示各角色职责与交互
- **状态机**：复杂业务状态流转图

#### 2.2 流程设计原则
- **最小步骤**：流程节点越少越好
- **角色清晰**：每个节点由谁负责明确
- **异常处理**：每个节点可能出什么问题？怎么处理？
- **可回溯**：用户操作可撤销、可查看历史

#### 2.3 流程图规范
- **开始/结束**：明确的入口和出口
- **角色泳道**：跨部门流程必须用泳道图
- **异常分支**：所有异常情况都要有处理路径
- **决策节点**：有分支的地方要有明确判断条件

### 使用工具
- Write：使用 Mermaid 语法生成流程图

### 工作流程
1. 梳理业务场景
2. 确定参与角色
3. 绘制主流程
4. 补充分支流程
5. 设计异常处理
6. 评审优化

### 输出标准
- 业务流程图.md
  - 泳道图（跨部门流程）
  - 状态机图（复杂状态流转）
  - 流程说明（每个节点的详细说明）
  - 异常处理说明

## 技能4：功能清单编写

### 技能描述
将需求转化为可执行的功能清单，为开发提供清晰的指导。

### 具体能力

#### 3.1 功能拆解
- **MECE原则**：相互独立，完全穷尽
- **颗粒度控制**：一个功能点可在一周内完成
- **依赖关系**：明确功能之间的依赖顺序

#### 3.2 功能描述规范
- **功能名称**：动宾结构，如"查看用户列表"
- **功能描述**：做什么？为谁做？做到什么程度？
- **业务规则**：什么条件下可以操作？什么情况不允许？
- **前置条件**：操作前需要满足什么？
- **后置条件**：操作后会改变什么？

#### 3.3 验收标准
- **可测试**：能用自动化脚本验证
- **可观测**：结果能用肉眼看清楚
- **边界条件**：正常、异常、边界三种情况

### 使用工具
- Kaiwu Form MCP：功能清单管理
- Write：生成功能清单文档

### 工作流程
1. 拆解需求为功能点
2. 编写功能描述
3. 定义业务规则
4. 编写验收标准
5. 标注依赖关系
6. 评审确认

### 输出标准
> 说明：一级目录=一级导航菜单；二级目录=二级导航菜单（若无，则写"/"）

- 功能清单.md
  - ID（如 F01）
  - 一级目录（一级导航菜单）
  - 二级目录（二级导航菜单，无则写"/"）
  - 页面名称
  - 详细描述
  - 优先级（P0/P1/P2/P3）
  - 复杂度（高/中/低）
  - 依赖关系（依赖的功能ID）
  - 验收标准

## 技能5：React+组件库开发（融合20年+ UI/UX设计经验）

### 技能描述
结合20年+资深UI/UX设计经验，使用React+Ant Design组件库开发高可用、高体验的企业级页面。不仅实现功能，更注重用户体验、视觉设计和交互细节。

### 具体能力

#### 4.1 用户体验设计（UX）
- **用户研究**
  - 用户画像分析：目标用户特征、使用场景、痛点需求
  - 用户旅程地图：梳理用户完整操作流程
  - 可用性原则： Nielsen十大可用性原则应用

- **信息架构**
  - 内容组织：清晰的信息层级和分类
  - 导航设计：面包屑、菜单、标签页等导航模式
  - 搜索设计：搜索框位置、筛选条件、结果展示

- **交互设计**
  - 操作反馈：即时反馈、状态提示、错误处理
  - 防错设计：确认对话框、撤销操作、默认值
  - 效率优化：快捷键、批量操作、智能填充

#### 4.2 视觉设计（UI）
- **设计系统**
  - 色彩系统：主色、辅助色、功能色、中性色
  - 字体系统：字号、字重、行高、字间距
  - 间距系统：8px网格系统、内边距、外边距
  - 阴影系统：层级阴影、悬浮效果

- **组件规范**
  - 按钮规范：主按钮、次按钮、文字按钮、图标按钮
  - 表单规范：输入框、选择器、日期选择、上传
  - 表格规范：列宽、排序、筛选、分页、操作列
  - 卡片规范：信息密度、图片比例、操作区域

- **视觉层次**
  - 重要性排序：通过大小、颜色、位置区分重要性
  - 视觉引导：F型浏览、Z型浏览模式应用
  - 留白艺术：呼吸感、分组感、焦点突出

#### 4.3 页面类型与设计模式
- **管理后台页面**
  - Dashboard设计：数据可视化、关键指标、快捷入口
  - 列表页设计：搜索筛选区、数据表格、批量操作
  - 表单页设计：分步表单、分组表单、校验反馈
  - 详情页设计：信息展示、关联操作、历史记录

- **响应式设计**
  - 断点设置：xs/sm/md/lg/xl断点定义
  - 布局适配：侧边栏折叠、表格横向滚动、卡片换行
  - 触控优化：按钮大小、手势操作、移动端导航

#### 4.4 React+Ant Design开发技巧
- **组件选型**
  - 基础组件：Button、Input、Select、DatePicker等
  - 布局组件：Row、Col、Space、Divider等
  - 数据展示：Table、List、Card、Descriptions等
  - 反馈组件：Modal、Drawer、Message、Notification等

- **状态管理**
  - useState：组件内部状态
  - useEffect：副作用处理、数据获取
  - useContext：跨组件状态共享
  - 自定义Hooks：逻辑复用

- **性能优化**
  - React.memo：组件缓存
  - useMemo/useCallback：计算缓存
  - 虚拟滚动：大数据量列表优化
  - 懒加载：图片、组件按需加载

- **代码规范**
  - 组件命名：PascalCase，语义化命名
  - 文件组织：按功能模块划分
  - 样式管理：CSS Modules或Styled Components
  - 类型定义：PropTypes或TypeScript

- **字段设计规范（重要）**
  - **业务关联性校验**：页面中的每个字段必须与业务流程中其他页面的字段建立关联
    - 上游页面输出字段 → 当前页面输入字段
    - 当前页面输出字段 → 下游页面输入字段
    - 避免孤立字段：每个字段都必须在业务流程中有明确的来源和去向
  - **字段溯源检查**：
    - 这个字段的数据从哪里来？（上游页面/外部系统/用户输入）
    - 这个字段的数据到哪里去？（下游页面/数据存储/其他系统）
    - 如果字段没有明确的上下游关联，需要重新评估是否有必要
  - **流程一致性校验**：
    - 确保字段命名在不同页面保持一致
    - 确保字段数据类型在不同页面保持一致
    - 确保字段业务含义在不同页面保持一致
  - **字段必要性评估**：
    - 这个字段对业务流程是否必要？
    - 去掉这个字段是否影响业务完整性？
    - 是否存在冗余字段可以合并或删除？

### 使用工具
- React：前端框架
- Ant Design：企业级React组件库
- Figma/Sketch：UI设计工具（设计稿参考）
- Chrome DevTools：调试与性能分析
- Write：生成React组件代码

### 工作流程
1. **需求理解**：理解业务需求，明确用户目标和场景
2. **选择页面类型**：确定页面类型（列表/表单/详情）
3. **UX设计**：用户旅程地图、信息架构、交互流程
4. **UI设计**：视觉稿设计、设计系统应用、组件选型
5. **React开发**：组件开发、状态管理、交互实现
6. **体验优化**：可用性测试、性能优化、细节打磨
7. **评审迭代**：设计评审、用户反馈、持续优化

### 输出标准
- React页面代码（.jsx/.tsx）
  - 符合Ant Design设计规范
  - 良好的用户体验和视觉设计
  - 清晰的组件结构和代码注释
  - 完整的交互逻辑和状态管理
- UI设计说明
  - 设计思路说明
  - 关键设计决策
  - 用户体验优化点

## 技能6：PRD文档编写

### 技能描述
编写完整、清晰的PRD文档，作为需求的核心输出物。

### 具体能力

#### 5.1 PRD结构
- **文档封面**：版本号、编写人、日期
- **修订记录**：每次修改的变更说明
- **目录**：快速导航

- **需求背景**
  - 需求来源（谁提的？）
  - 业务背景（为什么现在要做？）
  - 解决什么问题？（痛点描述）
  - 预期收益（做成什么样？）

- **功能概述**
  - 功能范围（做什么？不做什么？）
  - 用户画像（谁用？）
  - 使用场景（什么时候用？）
  - 业务流程图

- **详细功能描述**
  - 功能1：功能描述、业务规则、流程图、React页面截图、验收标准
  - 功能2：...
  - 功能N：...

- **非功能需求**
  - 性能要求（响应时间？并发量？）
  - 安全要求（权限？数据加密？）
  - 兼容性要求（浏览器？系统版本？）

- **测试要点**
  - 功能测试点
  - 性能测试点
  - 安全测试点

- **上线方案**
  - 数据迁移
  - 回滚方案
  - 监控指标

#### 5.2 PRD写作原则
- **以用户为中心**：描述用户能做什么，不是系统有什么功能
- **清晰无歧义**：每个规则都要描述清楚
- **可执行**：开发能看懂怎么实现，测试能看懂怎么验证
- **版本管理**：有变更要记录，方便追溯

### 使用工具
- Read：读取PRD模板
- Write：生成PRD文档

### 工作流程
1. 收集前期输出物（需求分析、流程图、功能清单、原型）
2. 编写需求背景
3. 编写功能概述
4. 逐个编写详细功能
5. 编写非功能需求
6. 编写测试要点
7. 评审修订

### 输出标准
- PRD文档.md
  - 需求背景
  - 功能概述（含业务流程图）
  - 详细功能描述（含React页面截图和UI设计说明）
  - 非功能需求
  - 测试要点
  - 上线方案

## 技能7：需求评审与项目管理

### 技能描述
组织需求评审，协调各方资源，推动需求落地。

### 具体能力

#### 6.1 需求评审
- **评审前准备**
  - PRD提前发送
  - 评审会议程
  - 参会人员确认

- **评审进行**
  - 需求讲解（产品经理）
  - 技术可行性评估（开发）
  - 测试可行性评估（测试）
  - 设计可行性评估（UI/UX）

- **评审后跟进**
  - 记录问题与决策
  - 更新需求文档
  - 跟踪问题解决

#### 6.2 项目管理
- **版本规划**：确定每个版本的范围
- **进度跟踪**：了解开发/测试进度
- **风险预警**：及时发现并上报风险
- **需求变更**：变更流程与记录

#### 6.3 5S敏捷开发实践
将5S管理理念融入项目管理，提升团队效率：
- **整理（Seiri）**：聚焦核心需求，清理低优先级任务，保持需求池整洁
- **整顿（Seiton）**：标准化文档模板、规范会议流程、建立知识库
- **清扫（Seiso）**：定期清理技术债务，及时修复Bug，保持代码健康
- **清洁（Seiketsu）**：建立Code Review机制，保持文档同步，维护环境一致性
- **素养（Shitsuke）**：团队持续学习，项目复盘总结，培养质量意识

### 工作流程
1. 准备评审材料
2. 组织评审会议
3. 记录评审意见
4. 跟进问题解决
5. 更新需求文档

### 输出标准
- 需求评审会议纪要.md
  - 会议基本信息（时间、地点、参会人员）
  - 评审内容概述（评审的功能点/需求）
  - 评审意见汇总（各方意见、问题、建议）
  - 决策结论（通过/不通过/需修改）
  - 待办事项（责任人、完成时间）
- 项目进度跟踪表.md
  - 版本规划（各版本范围、时间节点）
  - 任务状态（未开始/进行中/已完成）
  - 风险清单（风险描述、影响程度、应对措施）
  - 变更记录（变更内容、原因、影响）
