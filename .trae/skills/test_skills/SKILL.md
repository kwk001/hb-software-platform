# 测试工程师专业技能文档

## 技能0：测试需求分析（核心能力）

### 技能描述
这是20年资深测试专家最核心的能力——在动手测试之前，先深入理解需求，制定合适的测试策略。

### 具体能力

#### 0.1 需求理解与澄清
- **需求文档评审**：参与PRD评审，理解功能逻辑
- **需求疑问澄清**：发现需求描述不清、逻辑矛盾的地方
- **测试范围界定**：明确哪些要测、哪些不测
- **验收标准确认**：明确功能通过的标准

#### 0.2 测试策略制定
- **测试类型选择**
  - 功能测试：验证功能是否符合需求
  - 界面测试：验证UI是否符合设计稿
  - 接口测试：验证API接口正确性
  - 性能测试：验证系统性能指标
  - 安全测试：验证系统安全性
  - 兼容性测试：验证多平台兼容性
  - 回归测试：验证修改未影响其他功能

- **测试方法选择**
  - 手工测试 vs 自动化测试
  - 黑盒测试 vs 白盒测试
  - 探索性测试 vs 脚本化测试

#### 0.3 测试优先级
- **P0（核心功能）**：必须100%通过，否则不能上线
- **P1（重要功能）**：主要功能，允许少量非阻塞问题
- **P2（一般功能）**：辅助功能，允许存在一定问题
- **P3（边缘功能）**：锦上添花，可延后修复

#### 0.4 风险评估
- **高风险区域**：复杂逻辑、核心流程、历史Bug多发区
- **测试重点**：优先覆盖高风险区域
- **资源分配**：根据风险调整测试投入

#### 0.5 测试环境准备
- **环境搭建**：测试服务器、数据库、中间件
- **测试数据准备**：基础数据、边界数据、异常数据
- **工具准备**：缺陷管理工具、自动化测试工具

### 使用工具
- Read：读取PRD文档
- Kaiwu Form MCP：测试需求管理
- Write：生成测试计划

### 工作流程
1. 参与需求评审
2. 理解功能逻辑
3. 澄清需求疑问
4. 制定测试策略
5. 确定测试优先级
6. 准备测试环境

### 输出标准
- 测试计划.md
  - 测试范围（测什么？不测什么？）
  - 测试策略（用什么方法测？）
  - 测试优先级（P0/P1/P2/P3）
  - 测试环境（用什么环境测？）
  - 测试进度（什么时候完成？）
  - 风险评估（有什么风险？）

## 技能1：测试用例设计

### 技能描述
设计全面、有效的测试用例，覆盖正常场景、异常场景和边界场景。

### 具体能力

#### 1.1 测试用例设计方法
- **等价类划分法**
  - 有效等价类：符合要求的输入
  - 无效等价类：不符合要求的输入
  - 每个等价类选一个代表测试

- **边界值分析法**
  - 边界值：最小值、最大值、刚好小于最小值、刚好大于最大值
  - 边界附近最容易出问题

- **因果图法**
  - 分析输入条件与输出结果之间的因果关系
  - 适用于多条件组合场景

- **状态转换法**
  - 分析系统状态之间的转换
  - 适用于有状态的业务流程

- **场景法**
  - 基于业务流程设计测试场景
  - 基本流 + 备选流

- **正交试验法**
  - 多因素多水平的最优组合
  - 用最少的用例覆盖最多的组合

#### 1.2 测试用例要素
- **用例编号**：唯一标识（如 TC-001）
- **用例标题**：简洁描述测试内容
- **前置条件**：执行用例前需要满足的条件
- **测试步骤**：详细的操作步骤
- **预期结果**：期望的输出或行为
- **优先级**：P0/P1/P2/P3
- **所属模块**：功能模块归属
- **测试类型**：功能/接口/性能/兼容等

#### 1.3 测试用例设计原则
- **独立性**：每个用例独立，不依赖其他用例
- **可重复性**：多次执行结果一致
- **可判定性**：结果明确，通过/失败清晰
- **完整性**：覆盖所有需求点
- **经济性**：避免冗余，用例数量适中

### 使用工具
- Excel/Write：编写测试用例
- Kaiwu Form MCP：测试用例管理

### 工作流程
1. 分析需求功能点
2. 选择设计方法
3. 编写测试用例
4. 评审用例完整性
5. 维护用例更新

### 输出标准
- 测试用例.xlsx/.md
  - 用例编号、标题、前置条件、测试步骤、预期结果
  - 优先级、所属模块、测试类型
  - 覆盖所有需求点（需求-用例追溯矩阵）

## 技能2：功能测试执行

### 技能描述
按照测试用例执行测试，发现并记录缺陷，跟踪缺陷修复。

### 具体能力

#### 2.1 测试执行流程
- **冒烟测试**：验证基本功能可用，版本可测
- **详细测试**：按照用例逐条执行
- **探索性测试**：自由测试，发现用例未覆盖的问题
- **回归测试**：验证Bug修复，确认未引入新问题

#### 2.2 缺陷管理
- **缺陷记录要素**
  - 缺陷标题：简洁描述问题
  - 严重程度：致命/严重/一般/轻微
  - 优先级：紧急/高/中/低
  - 复现步骤：详细操作步骤
  - 实际结果：实际表现
  - 预期结果：期望表现
  - 截图/录屏：直观展示问题
  - 环境信息：浏览器、系统版本等

- **缺陷生命周期**
  - 新建 → 确认 → 分配 → 修复 → 验证 → 关闭
  - 重新打开（验证不通过）
  - 拒绝（不是缺陷）
  - 延期（后续版本修复）

- **缺陷跟踪**
  - 每日跟踪缺陷修复进度
  - 验证修复结果
  - 统计缺陷数据（发现趋势、修复率等）

#### 2.3 测试数据管理
- **测试数据准备**：基础数据、边界数据、异常数据
- **数据清理**：测试后清理，避免影响其他测试
- **数据备份**：重要测试数据备份

### 使用工具
- Chrome DevTools MCP：Web测试
- Kaiwu Form MCP：缺陷管理
- Write：生成测试记录

### 工作流程
1. 执行冒烟测试
2. 按用例执行详细测试
3. 记录发现的缺陷
4. 进行探索性测试
5. 跟踪缺陷修复
6. 执行回归测试

### 输出标准
- 测试执行记录
  - 执行的用例列表
  - 通过/失败状态
  - 发现的缺陷列表
  - 测试覆盖率统计

## 技能3：接口测试

### 技能描述
测试系统接口的正确性、稳定性和性能。

### 具体能力

#### 3.1 接口分析
- **接口文档评审**：URL、Method、参数、返回值
- **接口依赖分析**：接口之间的调用关系
- **接口安全分析**：鉴权、加密、防重放等

#### 3.2 接口测试用例设计
- **正常场景**：正确参数，验证返回结果
- **异常场景**：错误参数，验证错误处理
- **边界场景**：参数边界值测试
- **安全场景**：未授权访问、SQL注入等
- **性能场景**：并发、响应时间等

#### 3.3 接口测试执行
- **单接口测试**：验证单个接口功能
- **接口链测试**：验证接口调用链
- **数据一致性**：验证数据库数据正确性
- **日志验证**：验证接口日志记录

### 使用工具
- Postman：接口测试工具
- JMeter：接口性能测试
- Write：生成接口测试脚本

### 工作流程
1. 获取接口文档
2. 分析接口逻辑
3. 设计测试用例
4. 执行接口测试
5. 验证返回结果
6. 记录测试结果

### 输出标准
- 接口测试用例
  - 接口URL、Method
  - 请求参数（正常/异常/边界）
  - 预期返回结果
  - 实际返回结果
  - 测试结论

## 技能4：性能测试

### 技能描述
验证系统在高负载下的性能表现，发现性能瓶颈。

### 具体能力

#### 4.1 性能指标
- **响应时间**：请求到响应的时间
- **吞吐量**：单位时间处理的请求数
- **并发用户数**：同时在线用户数
- **资源利用率**：CPU、内存、磁盘、网络
- **错误率**：请求失败的比例

#### 4.2 性能测试类型
- **负载测试**：逐步增加负载，找到性能拐点
- **压力测试**：超过正常负载，测试系统极限
- **稳定性测试**：长时间运行，测试系统稳定性
- **峰值测试**：模拟突发流量，测试系统应对能力

#### 4.3 性能测试执行
- **测试场景设计**：模拟真实用户行为
- **测试数据准备**：足够的数据量
- **测试脚本编写**：录制或编写测试脚本
- **测试执行**：逐步增加负载
- **结果分析**：找出性能瓶颈

### 使用工具
- JMeter：性能测试工具
- LoadRunner：企业级性能测试
- Write：生成性能测试报告

### 工作流程
1. 确定性能指标
2. 设计测试场景
3. 准备测试数据
4. 编写测试脚本
5. 执行性能测试
6. 分析测试结果

### 输出标准
- 性能测试报告
  - 测试场景描述
  - 性能指标数据
  - 性能瓶颈分析
  - 优化建议

## 技能5：兼容性测试

### 技能描述
验证系统在不同环境下的兼容性。

### 具体能力

#### 5.1 浏览器兼容性
- **主流浏览器**：Chrome、Firefox、Safari、Edge
- **浏览器版本**：最新版、主流旧版
- **测试内容**：页面显示、功能操作、样式兼容性

#### 5.2 操作系统兼容性
- **桌面系统**：Windows、macOS、Linux
- **移动系统**：iOS、Android
- **测试内容**：安装、运行、功能、性能

#### 5.3 移动端兼容性
- **屏幕尺寸**：不同分辨率适配
- **设备类型**：手机、平板
- **测试内容**：触摸操作、横竖屏、手势

#### 5.4 兼容性测试策略
- **矩阵法**：浏览器 × 操作系统组合
- **优先级法**：优先测试主流组合
- **自动化法**：使用自动化工具批量测试

### 使用工具
- BrowserStack：云兼容性测试
- Chrome DevTools MCP：设备模拟
- Write：生成兼容性测试报告

### 工作流程
1. 确定兼容性范围
2. 选择测试环境
3. 设计测试用例
4. 执行兼容性测试
5. 记录兼容性问题
6. 生成测试报告

### 输出标准
- 兼容性测试报告
  - 测试环境矩阵
  - 兼容性测试结果
  - 不兼容问题列表
  - 修复建议

## 技能6：测试报告编写

### 技能描述
编写全面、清晰的测试报告，为项目决策提供依据。

### 具体能力

#### 6.1 测试报告结构
- **报告封面**：项目名称、版本、日期、编写人
- **测试概况**：测试范围、测试时间、测试人员
- **测试执行统计**：用例总数、通过数、失败数、通过率
- **缺陷统计**：缺陷总数、致命/严重/一般/轻微分布
- **缺陷趋势**：每日发现/修复趋势图
- **遗留问题**：未修复问题及影响评估
- **测试结论**：是否达到上线标准
- **发布建议**：建议发布/延期/修复后发布
- **风险说明**：已知风险及应对措施

#### 6.2 测试报告内容
- **测试范围**：测了什么？没测什么？
- **测试环境**：用什么环境测的？
- **测试用例执行**：执行了多少？通过率多少？
- **缺陷分析**：发现了多少？严重程度分布？
- **性能数据**：响应时间、吞吐量等
- **兼容性结果**：支持哪些环境？
- **遗留问题**：还有什么问题没解决？
- **上线建议**：能不能上线？有什么风险？

#### 6.3 测试报告原则
- **客观真实**：如实反映测试结果
- **数据支撑**：用数据说话
- **重点突出**：关键问题重点说明
- **建议明确**：给出明确的上线建议

### 使用工具
- Write：编写测试报告
- Excel：生成图表

### 工作流程
1. 收集测试数据
2. 统计测试指标
3. 分析测试结果
4. 编写测试报告
5. 评审报告内容
6. 发布测试报告

### 输出标准
- 测试报告.md
  - 测试概况
  - 测试执行统计
  - 缺陷统计与分析
  - 性能测试结果
  - 兼容性测试结果
  - 遗留问题说明
  - 测试结论
  - 发布建议
