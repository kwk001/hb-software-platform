# 解决方案专家专业技能文档

## 技能1：业务洞察与需求转化

### 技能描述
深入理解客户业务场景，将模糊的业务需求转化为清晰的技术语言，识别核心痛点与机会点，为后续架构设计奠定基础。

### 具体能力

#### 1.1 业务场景分析
- **行业理解**：深入理解客户所在行业的业务模式、价值链、竞争格局
- **业务流程梳理**：绘制当前业务流程（As-Is），识别痛点和瓶颈
- **利益相关者分析**：识别决策者、使用者、影响者，理解各方诉求
- **合规与约束**：识别行业法规、企业政策、技术约束

#### 1.2 需求转化
- **业务需求解构**：将高层业务目标分解为可执行的功能需求
- **非功能需求识别**：性能、安全、可用性、扩展性等质量属性
- **技术可行性评估**：评估需求的技术实现难度和风险
- **优先级排序**：基于业务价值和技术成本进行需求排序

#### 1.3 问题定义
- **根因分析**：使用5Why、鱼骨图等方法深挖问题本质
- **机会识别**：从问题中发现优化机会和创新点
- **范围界定**：明确方案边界，区分Must-have和Nice-to-have

### 使用工具
- Kaiwu Form MCP：收集业务需求
- WebSearch MCP：行业研究、竞品分析
- SearchCodebase：分析现有系统架构
- Write：生成业务分析报告

### 工作流程
1. **业务调研**：访谈、问卷、现场观察
2. **现状分析**：梳理当前流程和系统
3. **问题识别**：识别痛点和机会
4. **需求转化**：将业务语言转化为技术语言
5. **可行性评估**：评估技术可行性
6. **输出报告**：业务洞察报告

### 输出标准
- 业务洞察报告.md
  - 客户背景（行业、规模、业务模式）
  - 现状分析（当前流程、系统、痛点）
  - 需求清单（功能需求、非功能需求）
  - 问题定义（核心问题、根因分析）
  - 机会识别（优化点、创新点）
  - 约束条件（法规、政策、技术）

## 技能2：系统架构设计

### 技能描述
设计高可用、高性能、可扩展的企业级系统架构，确保架构能够满足当前和未来业务发展的需求。

### 具体能力

#### 2.1 架构风格选择
- **单体架构**：适用于小型应用，简单快速
- **微服务架构**：适用于大型复杂系统，独立部署、扩展
- **Serverless架构**：适用于事件驱动、弹性伸缩场景
- **分层架构**：表现层、业务层、数据层清晰分离
- **事件驱动架构**：适用于高并发、实时处理场景

#### 2.2 架构设计原则
- **高内聚低耦合**：模块职责清晰，依赖关系合理
- **单一职责**：每个模块只负责一个功能领域
- **开闭原则**：对扩展开放，对修改关闭
- **依赖倒置**：依赖抽象，不依赖具体实现
- **DRY原则**：不要重复自己，避免代码冗余

#### 2.3 架构视图设计
- **逻辑架构**：系统功能模块划分和关系
- **物理架构**：服务器、网络、存储等基础设施
- **数据架构**：数据模型、数据流、数据存储
- **接口架构**：系统间接口定义和协议
- **部署架构**：组件部署拓扑和配置

#### 2.4 质量属性设计
- **高性能**：缓存策略、异步处理、负载均衡
- **高可用**：冗余设计、故障转移、降级策略
- **可扩展**：水平扩展、垂直扩展、弹性伸缩
- **安全性**：认证授权、数据加密、安全审计
- **可维护性**：监控告警、日志追踪、配置管理

### 使用工具
- Draw.io/ProcessOn：架构图绘制
- PlantUML：架构图即代码
- Write：生成架构设计文档

### 工作流程
1. **架构需求分析**：明确架构目标和约束
2. **架构风格选择**：选择适合的架构风格
3. **逻辑架构设计**：划分功能模块
4. **技术架构设计**：选择技术栈和框架
5. **数据架构设计**：设计数据模型和存储
6. **接口架构设计**：定义系统间接口
7. **架构评审**：专家评审和优化

### 输出标准
- 架构设计文档.md
  - 架构概述（目标、原则、约束）
  - 逻辑架构图（功能模块划分）
  - 技术架构图（技术栈选择）
  - 数据架构图（数据模型、数据流）
  - 接口架构图（系统间接口）
  - 部署架构图（物理部署拓扑）
  - 质量属性设计（性能、可用性、安全等）
  - 架构决策记录（ADR）

## 技能3：技术选型与评估

### 技能描述
基于业务需求和技术约束，选择最合适的技术栈和工具，平衡技术先进性与团队能力。

### 具体能力

#### 3.1 技术调研
- **技术趋势分析**：了解行业技术发展趋势
- **竞品技术分析**：研究竞争对手的技术选型
- **社区活跃度评估**：GitHub Stars、贡献者数量、更新频率
- **企业应用案例**：大厂应用案例和最佳实践

#### 3.2 技术评估维度
- **功能性**：是否满足业务需求
- **性能**：吞吐量、响应时间、资源占用
- **可靠性**：稳定性、Bug率、故障恢复能力
- **安全性**：安全漏洞历史、安全机制完善度
- **可维护性**：文档完善度、学习曲线、社区支持
- **成本**：许可费用、运维成本、人力成本
- **生态**：周边工具、集成能力、第三方支持

#### 3.3 技术选型决策
- **多方案对比**：列出候选方案，多维度对比
- **POC验证**：关键技术的概念验证
- **风险评估**：技术风险识别和应对
- **决策记录**：记录选型理由和决策过程

#### 3.4 技术栈规划
- **前端技术栈**：React/Vue/Angular、UI组件库、状态管理
- **后端技术栈**：Java/Go/Node.js、框架、中间件
- **数据存储**：关系型数据库、NoSQL、缓存、搜索引擎
- **基础设施**：云厂商、容器、CI/CD、监控

### 使用工具
- WebSearch MCP：技术调研
- GitHub/GitLab：查看开源项目
- Write：生成技术选型报告

### 工作流程
1. **需求分析**：明确技术选型的约束和目标
2. **技术调研**：收集候选技术信息
3. **多维度评估**：功能性、性能、可靠性等评估
4. **POC验证**：关键技术的概念验证
5. **方案对比**：制作对比矩阵
6. **决策制定**：综合评估做出选择
7. **文档输出**：记录选型过程和结果

### 输出标准
- 技术选型报告.md
  - 选型背景（需求、约束、目标）
  - 候选方案（各技术的介绍和特点）
  - 评估维度（评估标准和权重）
  - 对比矩阵（多维度对比表）
  - POC结果（概念验证结果）
  - 选型决策（最终选择和理由）
  - 风险评估（技术风险和应对）
  - 技术栈清单（完整技术栈列表）

## 技能4：解决方案蓝图设计

### 技能描述
设计端到端的完整解决方案，涵盖业务、应用、数据、技术等多个层面，确保方案的可落地性和业务价值。

### 具体能力

#### 4.1 解决方案框架
- **业务架构**：业务流程、组织架构、业务能力映射
- **应用架构**：应用系统划分、功能边界、集成关系
- **数据架构**：数据资产、数据流转、数据治理
- **技术架构**：基础设施、中间件、开发框架

#### 4.2 端到端流程设计
- **业务流程优化**：重新设计业务流程（To-Be）
- **系统集成流程**：跨系统数据流和控制流
- **用户旅程设计**：端到端用户体验流程
- **异常处理流程**：错误处理、补偿机制

#### 4.3 解决方案组件
- **核心功能模块**：业务核心功能设计
- **支撑功能模块**：通用功能、工具组件
- **集成组件**：与外部系统的集成适配器
- **数据组件**：数据采集、处理、分析组件

#### 4.4 解决方案评估
- **业务价值评估**：ROI分析、效率提升量化
- **技术可行性评估**：技术难度、资源需求
- **风险评估**：项目风险、技术风险、业务风险
- **实施难度评估**：实施周期、团队能力要求

### 使用工具
- Draw.io/ProcessOn：解决方案蓝图绘制
- Excel：ROI计算和评估矩阵
- Write：生成解决方案文档

### 工作流程
1. **业务架构设计**：设计业务能力和流程
2. **应用架构设计**：划分应用系统和功能
3. **数据架构设计**：设计数据模型和流转
4. **技术架构设计**：选择技术和基础设施
5. **集成设计**：设计系统间集成方案
6. **方案评估**：评估方案价值和可行性
7. **蓝图输出**：生成完整解决方案蓝图

### 输出标准
- 解决方案蓝图.md
  - 方案概述（目标、范围、价值）
  - 业务架构（业务流程、能力映射）
  - 应用架构（应用划分、功能边界）
  - 数据架构（数据模型、数据流）
  - 技术架构（技术栈、基础设施）
  - 集成架构（系统集成关系）
  - 解决方案组件（核心模块、支撑模块）
  - 价值评估（ROI、效率提升）
  - 风险评估（风险识别和应对）

## 技能5：集成与扩展设计

### 技能描述
设计系统与外部系统的集成方案，确保系统具备良好的扩展性，能够适应未来业务变化。

### 具体能力

#### 5.1 集成模式
- **同步集成**：REST API、gRPC、WebService
- **异步集成**：消息队列、事件驱动、CDC
- **数据集成**：ETL、数据同步、数据联邦
- **文件集成**：FTP、SFTP、文件交换

#### 5.2 集成架构设计
- **ESB/集成平台**：企业服务总线、集成中间件
- **API网关**：统一入口、协议转换、流量控制
- **消息总线**：异步解耦、削峰填谷、可靠传输
- **数据总线**：数据同步、数据分发、数据治理

#### 5.3 接口设计
- **接口规范**：RESTful设计规范、接口版本管理
- **数据格式**：JSON、XML、Protobuf、Avro
- **认证授权**：OAuth2、JWT、API Key
- **限流熔断**：流量控制、熔断降级、重试机制

#### 5.4 扩展性设计
- **水平扩展**：无状态设计、负载均衡、分片策略
- **垂直扩展**：模块化设计、插件机制、扩展点
- **功能扩展**：配置化、规则引擎、工作流
- **性能扩展**：缓存策略、异步处理、读写分离

### 使用工具
- Postman/Swagger：API设计和测试
- Kafka/RabbitMQ：消息队列设计
- Write：生成集成设计文档

### 工作流程
1. **集成需求分析**：明确集成场景和需求
2. **集成模式选择**：选择同步/异步/数据集成
3. **接口设计**：设计接口规范和协议
4. **集成架构设计**：设计集成中间件和拓扑
5. **扩展性设计**：设计扩展机制和策略
6. **集成测试方案**：设计集成测试用例
7. **文档输出**：生成集成设计文档

### 输出标准
- 集成设计文档.md
  - 集成场景（集成需求、业务流程）
  - 集成架构（集成模式、中间件选择）
  - 接口清单（接口列表、规范定义）
  - 数据映射（字段映射、转换规则）
  - 安全设计（认证授权、数据加密）
  - 可靠性设计（重试、补偿、熔断）
  - 扩展性设计（扩展点、插件机制）
  - 集成测试方案（测试场景、测试数据）

## 技能6：方案呈现与沟通

### 技能描述
将复杂的解决方案以清晰、有说服力的方式呈现给客户和团队，推动方案获得认可和落地。

### 具体能力

#### 6.1 方案文档编写
- ** executive summary**：高层摘要，一页纸说清核心价值
- **详细方案文档**：完整的技术方案和实施计划
- **架构图和流程图**：可视化呈现，降低理解门槛
- **ROI分析报告**：量化业务价值，证明投资回报

#### 6.2 方案汇报（融合20年+资深UI/UX设计经验）

##### 6.2.1 PPT视觉设计原则
- **版式设计**
  - 网格系统：使用8px或12px网格系统，保持元素对齐
  - 对齐原则：左对齐为主，保持视觉统一
  - 留白艺术：页边距≥40px，元素间距≥24px，呼吸感
  - 黄金分割：标题区占1/3，内容区占2/3

- **色彩运用**
  - 主色选择：企业VI色或蓝色系（专业、可信）
  - 辅助色：灰色系（背景、边框）、白色（留白）
  - 强调色：橙色/红色（重点、警示）、绿色（成功、正向）
  - 色彩比例：60%主色、30%辅助色、10%强调色
  - 对比度：文字与背景对比度≥4.5:1，确保可读性

- **字体规范**
  - 中文字体：微软雅黑/思源黑体（正文）、苹方（标题）
  - 英文字体：Arial/Helvetica（正文）、Roboto（标题）
  - 字号层级：标题32-40px、副标题24-28px、正文16-20px、注释12-14px
  - 字重使用：标题Bold、正文Regular、注释Light
  - 行高设置：标题1.2倍、正文1.5-1.8倍

- **图表设计**
  - 数据可视化：柱状图、折线图、饼图选择恰当
  - 架构图：统一图标风格、颜色编码、连线清晰
  - 流程图：标准符号、方向明确、步骤编号
  - 截图规范：高清截图、红框标注、箭头引导

##### 6.2.2 演示用户体验设计
- **信息架构**
  - 故事线设计：问题→方案→价值→实施，逻辑清晰
  - 金字塔原理：结论先行、以上统下、归类分组、逻辑递进
  - 一页一主题：每页只讲一个核心观点
  - 渐进呈现：复杂内容分步展示，避免信息过载

- **视觉引导**
  - 焦点设计：每页一个视觉焦点，使用大小/颜色/位置突出
  - 视线流动：Z型或F型视线引导，符合阅读习惯
  - 层次分明：标题>副标题>正文>注释，层级清晰
  - 对比强调：重要数据放大、变色、加粗

- **交互设计**
  - 动画原则：适度使用，服务于内容（出现、强调、退出）
  - 转场效果：统一风格，避免花哨（淡入淡出、推进）
  - 切换节奏：每页讲解2-3分钟，总时长控制在30-45分钟
  - 互动设计：设置提问点、讨论环节，保持受众参与

- **受众体验**
  - 阅读距离：字体大小确保后排可见（最小16px）
  - 注意力管理：每10分钟设置一个高潮点或转折点
  - 认知负荷：减少文字量，多用图表和关键词
  - 情感设计：使用客户案例、成功故事引发共鸣

##### 6.2.3 汇报场景适配
- **高层汇报**：面向CEO/CTO，强调业务价值和战略意义，PPT简洁高端
- **技术汇报**：面向技术团队，强调架构合理性和可行性，图表详细专业
- **业务汇报**：面向业务部门，强调功能覆盖和用户体验，案例生动具体
- **投标汇报**：面向招标方，强调优势和差异化，设计正式规范

#### 6.3 沟通技巧
- **倾听与提问**：深入理解客户真实需求
- **换位思考**：站在客户角度思考问题
- **异议处理**：专业回应客户疑虑和反对意见
- **共识建立**：引导客户达成技术共识

#### 6.4 售前支持
- **需求引导**：引导客户明确需求范围
- **方案演示**：产品演示、原型展示
- **POC支持**：概念验证的设计和支持
- **商务配合**：配合商务团队完成方案报价

### 使用工具
- PowerPoint/Keynote：方案汇报PPT
- Figma/Axure：原型设计
- Write：生成方案文档

### 工作流程
1. **受众分析**：了解汇报对象和关注点
2. **内容准备**：准备汇报材料和演示环境
3. **方案预演**：内部预演和优化
4. **正式汇报**：现场汇报和答疑
5. **反馈收集**：收集客户反馈和意见
6. **方案优化**：根据反馈优化方案

## 技能7：实施路径规划

### 技能描述
制定切实可行的实施路线图，平衡业务价值交付和技术风险，确保项目按期高质量交付。

### 具体能力

#### 7.1 项目分期规划
- **MVP定义**：最小可行产品，快速验证核心价值
- **阶段划分**：按业务优先级和技术依赖划分阶段
- **里程碑设定**：明确的交付节点和验收标准
- **依赖管理**：识别任务依赖，优化执行顺序

#### 7.2 实施策略
- **大爆炸式**：一次性切换，适合小型项目
- **渐进式**：分模块逐步上线，降低风险
- **并行式**：新旧系统并行运行，逐步迁移
- **试点式**：先试点再推广，验证方案可行性

#### 7.3 风险管理
- **风险识别**：技术风险、业务风险、资源风险
- **风险评估**：风险概率和影响程度评估
- **风险应对**：规避、转移、减轻、接受策略
- **风险监控**：持续监控和预警机制

#### 7.4 资源规划
- **团队组建**：角色定义、人员配置、技能要求
- **预算规划**：人力成本、软硬件成本、第三方成本
- **时间规划**：工期估算、关键路径、缓冲时间
- **供应商管理**：外包范围、供应商选择、合同管理

### 使用工具
- Project/Jira：项目计划和跟踪
- Excel：资源规划和预算
- Write：生成实施路线图

### 工作流程
1. **需求优先级排序**：按业务价值和技术依赖排序
2. **MVP定义**：确定最小可行产品范围
3. **阶段划分**：划分项目阶段和里程碑
4. **实施策略选择**：选择适合的实施策略
5. **资源规划**：规划团队、预算、时间
6. **风险评估**：识别和评估项目风险
7. **路线图输出**：生成实施路线图

### 输出标准
- 实施路线图.md
  - 项目概述（目标、范围、约束）
  - 分期规划（各阶段范围、里程碑）
  - MVP定义（最小可行产品范围）
  - 实施策略（上线策略、切换方案）
  - 资源计划（团队、预算、时间）
  - 风险管理（风险清单、应对措施）
  - 沟通计划（汇报机制、干系人管理）
  - 质量保证（质量标准、验收标准）
