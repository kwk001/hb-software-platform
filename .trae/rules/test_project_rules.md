## 任务执行规则

### 测试计划阶段任务
1. **需求分析与模式识别**：理解业务需求，识别测试点和验收标准，提取需求模式
2. **测试策略与知识自举**：确定测试类型、测试范围、测试方法，从历史测试经验中自动生成测试策略
3. **资源评估**：评估测试环境、测试数据、测试工具需求
4. **进度安排**：制定测试计划、里程碑、时间节点，设计2-4周的快速测试迭代计划
5. **风险评估**：识别测试风险、制定应对措施

### 测试设计阶段任务
1. **用例设计与模式识别**：使用等价类、边界值、场景法等设计测试用例，从需求和系统中识别测试模式
2. **用例评审**：组织用例评审，确保覆盖率和可执行性
3. **数据准备**：准备测试数据、测试环境、测试工具
4. **自动化规划**：识别可自动化场景，规划自动化脚本，基于历史测试经验自动生成测试用例

### 测试执行阶段任务
1. **冒烟测试**：验证核心流程，确保系统可测
2. **功能测试与即时总结**：按用例执行测试，记录测试结果，即时总结测试经验，提取测试模式
3. **缺陷管理与模式提取**：提交缺陷、跟踪修复、验证关闭，从缺陷中提取通用模式
4. **回归测试**：验证修复结果，确认影响范围
5. **测试报告与知识沉淀**：汇总测试结果，评估产品质量，沉淀测试知识，更新测试模式库

### 自动化测试阶段任务
1. **框架搭建**：选择合适的自动化测试框架
2. **脚本开发与自举**：编写自动化测试脚本，基于测试用例自动生成自动化脚本
3. **持续集成**：集成到 CI/CD 流水线
4. **报告生成**：配置自动化测试报告
5. **知识自举**：从自动化测试经验中自动生成新的测试知识和脚本

## MCP工具使用规则

- **使用场景**：执行 Web 应用的功能测试、UI 测试

## 测试规范

### 1. 测试用例规范

#### 用例结构
```
用例编号：TC-模块-序号（如：TC-LOGIN-001）
用例标题：简明描述测试目的
前置条件：执行测试前必须满足的条件
测试步骤：
  1. 步骤1
  2. 步骤2
  ...
预期结果：明确可验证的预期结果
优先级：P0/P1/P2/P3
测试模式：关联的测试模式（如：登录验证模式、边界值测试模式等）
```

#### 测试模式库建设
- 建立测试模式库，存储和管理通用测试模式
- 从测试用例中提取通用测试模式
- 定期更新和维护测试模式库
- 基于测试模式自动生成测试用例

#### 优先级定义
- **P0（阻塞级）**：核心功能，必须100%通过
- **P1（高优先级）**：重要功能，影响用户体验
- **P2（中优先级）**：一般功能，有替代方案
- **P3（低优先级）**：边缘场景，可延期修复

### 2. 缺陷管理规范

#### 缺陷等级
- **致命（Critical）**：系统崩溃、数据丢失、安全漏洞
- **严重（Major）**：核心功能不可用、主要流程阻塞
- **一般（Minor）**：非核心功能异常、有替代方案
- **轻微（Trivial）**：UI问题、文字错误、不影响功能

#### 缺陷生命周期
```
新建 → 已确认 → 处理中 → 已修复 → 待验证 → 已关闭
  ↓       ↓         ↓         ↓         ↓
  └───────┴─────────┴─────────┴─────────┘
         可重新打开（验证不通过）
```

### 3. 测试报告规范

#### 报告内容
- **测试概况**：测试范围、测试时间、测试人员
- **执行统计**：用例总数、通过数、失败数、阻塞数
- **缺陷统计**：缺陷总数、按等级分布、按模块分布
- **质量评估**：产品质量评价、发布建议
- **风险说明**：已知问题、遗留风险

## 自动化测试规范

### 1. UI自动化规范

#### 框架选择
- **Playwright**：推荐，支持多浏览器、自动等待、Trace Viewer
- **Cypress**：适合前端团队，实时重载、调试友好
- **Selenium**：生态成熟，支持多种语言

#### 脚本规范
- **Page Object模式**：页面对象与测试逻辑分离
- **数据驱动**：测试数据与脚本分离
- **异常处理**：添加try-catch，确保脚本健壮性
- **日志记录**：记录关键操作和断言结果

### 2. 接口自动化规范

#### 工具选择
- **Postman + Newman**：适合快速验证、团队协作
- **REST Assured**：Java生态，与项目集成好
- **JMeter**：适合性能测试和接口测试结合

#### 脚本规范
- **环境管理**：支持多环境切换（开发/测试/生产）
- **数据管理**：参数化、数据驱动、数据清理
- **断言设计**：状态码、响应体、响应时间、数据库
- **依赖管理**：处理接口依赖、数据依赖

## 协作规范

### 与产品经理协作
- **需求评审阶段**：识别测试点、评估可测性
- **测试计划阶段**：确认测试范围、优先级
- **缺陷跟踪阶段**：确认缺陷定级、修复优先级

### 与开发工程师协作
- **开发阶段**：提供可测性建议、准备测试数据
- **测试阶段**：沟通缺陷详情、协助定位问题
- **修复阶段**：验证修复结果、确认影响范围

### 与运维工程师协作
- **环境准备**：协助搭建测试环境
- **性能测试**：协助监控资源、分析瓶颈
- **发布阶段**：确认发布 checklist、验证线上功能
