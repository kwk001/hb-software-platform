## 需求优先级评估（ICE评分框架）

### 评分维度
| 维度 | 说明 | 评分范围 | 权重 |
|------|------|----------|------|
| **I**mpact（影响） | 对业务/用户的价值程度 | 1-10分 | 40% |
| **C**onfidence（信心） | 对需求可行性的把握 | 1-10分 | 30% |
| **E**ase（难度） | 实现的难易程度 | 1-10分 | 30% |

### 计算公式
```
ICE总分 = (Impact × 0.4 + Confidence × 0.3 + Ease × 0.3)

或简化版：
ICE总分 = (Impact × Confidence × Ease) / 100
```

### 决策标准
| 分数 | 建议 | 说明 | 处理时效 |
|------|------|------|----------|
| ≥7.0 | **立即处理** | 高价值、高信心、易实现 | 当天响应 |
| 5.0-7.0 | **排队处理** | 需要权衡资源 | 3天内响应 |
| <5.0 | **延后处理** | 风险高或价值低 | 一周后评估 |

### 评分示例
```
需求A：优化登录流程
- Impact: 8（影响所有用户）
- Confidence: 9（业界成熟方案）
- Ease: 8（改动范围小）
- ICE = 8×0.4 + 9×0.3 + 8×0.3 = 8.3 → 立即处理

需求B：重构整个架构
- Impact: 9（长期价值大）
- Confidence: 5（风险较高）
- Ease: 3（工作量巨大）
- ICE = 9×0.4 + 5×0.3 + 3×0.3 = 6.0 → 排队处理
```

## UI/UX设计质量检查规则（重要）

### 设计审查清单（必须逐项检查）
- **用户体验检查**
  - 是否从用户视角出发设计？
  - 操作流程是否简洁明了？
  - 是否有清晰的错误提示和反馈？
  - 是否考虑了异常场景和边界情况？
  - 是否符合Nielsen十大可用性原则？

- **视觉设计检查**
  - 色彩使用是否符合设计系统？
  - 字体大小、字重是否层次清晰？
  - 间距是否统一（8px网格系统）？
  - 视觉层次是否明确（重要信息突出）？
  - 是否有足够的留白（呼吸感）？

- **交互设计检查**
  - 每个操作是否有明确的反馈？
  - 按钮状态（默认、悬浮、点击、禁用）是否完整？
  - 表单校验是否即时、明确？
  - 加载状态、空状态、错误状态是否设计？
  - 是否有防错设计（确认对话框、撤销操作）？

- **响应式设计检查**
  - 是否适配不同屏幕尺寸？
  - 移动端触控区域是否足够大（最小44px）？
  - 表格在移动端是否有横向滚动或卡片化？
  - 侧边栏在移动端是否可折叠？

### 设计交付标准
- 必须提供设计说明文档
- 必须说明设计思路和决策依据
- 必须列出用户体验优化点
- 必须提供交互流程图（复杂功能）
- **必须记录设计质量评分**

## 需求-方案匹配验证循环规则（自动闭环机制）

### 循环位置
本验证循环发生在**阶段二：细化方案**内部，确保方案与需求高度匹配后再进入阶段三执行。

### 循环流程
```
需求输入 → 需求分析与总结 → 方案设计与实验 → 匹配验证与模式提取 → 是否匹配？
                                              ↓
                                    是 → 输出最终方案
                                    否 → 分析偏差 → 调整方案 → 再次验证
```

### 阶段定义

#### 阶段1：需求分析与总结（Analysis）
- 提取需求核心要点清单
- 识别功能边界和约束条件
- 明确验收标准和成功指标
- **计算ICE评分，确定优先级**
- **总结需求模式，提取通用需求类型**
- **识别高价值实验机会，制定实验计划**
- 输出：《需求要点清单》、《需求模式分析》

#### 阶段2：方案设计与实验（Design）
- 基于需求设计解决方案
- **设计实验方案，明确实验变量、指标和成功标准**
- **制定MVP交付计划，明确最小可用版本范围**
- **输出内容包括**：
  - React 开发的页面（UI 设计 + 交互实现）
  - 功能清单（详细功能点列表）
  - 业务流程图（Mermaid 流程图）
  - PRD 需求文档（完整需求规格说明）
  - 实验设计方案
  - MVP交付计划
- 输出：《方案设计文档》、《实验设计方案》

#### 阶段3：匹配验证与模式提取（Validation）
**验证条件检查项及权重**：
| 检查项 | 权重 | 评分标准 | 扣分规则 |
|--------|------|----------|----------|
| 功能覆盖度 | 25% | 方案功能是否覆盖所有需求功能点 | 每缺失1个功能点扣10% |
| 业务流程 | 20% | 方案流程是否符合需求描述 | 主流程错误扣20%，分支缺失扣10% |
| 边界条件 | 15% | 异常情况、边界场景是否处理 | 每遗漏1个场景扣5% |
| 验收标准 | 15% | 方案是否满足需求的成功指标 | 不满足扣15% |
| 约束条件 | 5% | 技术约束、业务约束是否满足 | 不满足扣5% |
| 用户体验 | 5% | 是否符合用户需求和期望 | 明显问题扣5% |
| 实验设计 | 10% | 实验方案是否完整、可行 | 不完整扣10% |
| 模式提取 | 5% | 是否提取通用模式并沉淀知识 | 未提取扣5% |

**匹配度计算公式**：
```
匹配度 = 100% - Σ(扣分项)

或加权计算：
匹配度 = Σ(各项得分 × 权重)
```

**匹配度评分标准**：
| 匹配度 | 状态 | 处理方式 |
|--------|------|----------|
| 100% | 🟢 完美 | 无需调整，直接通过 |
| 95%-99% | 🟢 优秀 | 轻微偏差，可选优化 |
| 90%-94% | 🟡 良好 | 建议调整，优化后通过 |
| 80%-89% | 🟠 及格 | 必须调整，重新验证 |
| <80% | 🔴 不及格 | 严重偏差，重新设计 |

#### 阶段4：偏差分析（Gap Analysis）
- 识别未覆盖的功能点
- 分析偏离需求的设计
- 记录遗漏的边界条件
- **量化偏差影响程度**
- 输出：《偏差分析报告》

#### 阶段5：方案调整（Adjustment）
- 针对偏差进行设计调整
- 补充缺失的功能点
- 修正偏离的业务流程
- **记录调整内容和原因**
- 输出：《调整后的方案》

### 循环控制规则

#### 终止条件（满足任一即终止）
1. **匹配度 ≥ 95%**：方案与需求高度匹配 ✅
2. **迭代次数 ≥ 5次**：防止无限循环 ⚠️
3. **连续2次无改进**：方案已无法优化 ⚠️
4. **用户人工确认**：用户认可当前方案 ✅

#### 最大迭代次数
- 默认：5次
- 复杂需求：可延长至8次
- 简单需求：可缩短至3次

#### 偏差分级处理
| 级别 | 偏差程度 | 处理方式 | 示例 | 扣分 |
|------|---------|---------|------|------|
| P0 | 核心功能缺失 | 必须调整 | 需求要求搜索功能，方案未设计 | -30% |
| P1 | 重要功能偏差 | 建议调整 | 需求说"快速查询"，方案未明确时效 | -15% |
| P2 | 优化建议 | 可选调整 | 建议增加导出功能，但需求未明确要求 | -5% |

### 输出要求

#### 每次循环必须输出
1. **当前迭代次数**：第 X 次验证
2. **匹配度百分比**：XX%（详细计算过程）
3. **ICE评分**：X分（I:X, C:X, E:X）
4. **偏差功能点清单**：
   - 未覆盖的功能点（P0/P1/P2）
   - 偏离需求的设计
   - 遗漏的边界条件
5. **模式提取**：本次迭代提取的通用模式
6. **知识沉淀**：本次迭代沉淀的可复用知识
7. **调整建议**：针对偏差的改进方案
8. **下次验证重点**：需要特别关注的检查项

#### 最终输出
- 《需求-方案匹配验证报告》（含ICE评分、匹配度计算）
- 《最终方案设计文档》
- 《偏差处理记录》（如有）
- 《实验日志》（项目复盘数据）
- 《模式提取报告》（本次迭代提取的通用模式）
- 《知识沉淀报告》（本次迭代沉淀的可复用知识）

### 执行规则

1. **自动触发**：需求设计完成后自动进入验证循环
2. **人工介入**：达到终止条件时询问用户确认
3. **记录存档**：每次循环结果保存到项目文档
4. **复盘总结**：循环结束后总结优化经验，更新实验日志

## 月度复盘规则

### 复盘频率
- **周度快速复盘**：每周五，15分钟
- **月度深度复盘**：每月最后一个工作日，1-2小时

### 复盘内容
1. **核心指标回顾**
   - 需求-方案匹配度达标率
   - 方案一次通过率
   - 用户满意度
   - 平均验证循环次数
   - 知识沉淀率
   - 实验成功率
   - 模式复用率

2. **漏斗分析**
   - 各阶段转化率
   - 最大流失点识别
   - 改进措施制定

3. **ICE评分准确性**
   - 预测准确率
   - 评分标准校准

4. **实验日志汇总**
   - 成功经验沉淀
   - 改进措施跟踪

5. **知识自举分析**
   - 知识自举率变化趋势
   - 自动产生知识的质量评估
   - 知识自举机制优化建议

6. **模式复用分析**
   - 模式复用率变化趋势
   - 高价值模式识别
   - 模式库优化建议

### 复盘输出
- 《月度复盘报告》
- 《改进行动计划》
- 《知识库更新》

## 不包含技术性设计内容
### 产品经理不应该设计的内容
- ❌ 后端接口设计（API接口定义、请求参数、响应格式）
- ❌ 数据库表结构设计（表名、字段名、字段类型、索引、关联关系）
- ❌ 具体技术实现方案（前端框架、后端框架、中间件）
- ❌ 技术选型（编程语言、数据库、缓存、消息队列）
- ❌ 性能优化方案（SQL优化、缓存策略、并发处理）
- ❌ 安全设计方案（权限控制、数据加密、防攻击）
- ❌ 部署架构设计（服务器配置、容器化、CI/CD）
