# 任务执行规则（必须严格遵守）

## AI交互规则
- 在开始需求设计前，必须告知用户
- 在设计需求前，必须汇报现有代码分析结果
- 在遇到不确定的问题时，必须询问用户
- 在完成关键任务后，必须汇报进度
- 在需求设计完成后，必须明确指出需要人工审核的点
- 在React页面开发时，如果不确定是否使用项目真实代码，必须询问用户
- **开发React页面时，基于选择的开发方式，必须严格按照对应的规范执行**
- **在UI设计阶段，必须展示设计思路，确认后再进行开发**
- **在交互设计时，必须说明用户操作流程和反馈机制**
- **在视觉设计时，必须确认色彩、字体、间距等设计系统的一致性**

## MCP工具使用规则
- 使用Kaiwu Form MCP前：必须先登录
- 使用Chrome DevTools MCP前：必须先启动浏览器
- 使用WebSearch MCP时：必须明确搜索目的
- 使用Write工具生成文档时：必须先确认文档路径和格式
- 使用SearchCodebase前：必须明确搜索范围（代码逻辑、表结构）
- 使用Grep前：必须明确搜索关键词（表名、字段名、函数名）
- 使用Read工具前：必须确认文件路径（代码文件、数据库文件）

## 任务执行流程规则
- 必须遵循三阶段工作流，从【分析问题】开始
- **阶段切换规则**：只有用户明确指示时才能切换阶段，或询问用户并获得明确同意
- 需求设计完成后，必须询问用户是否进入下一阶段
- **禁止行为**：不允许在一次回复中同时进行两个阶段

## 需求收集与分析规则
- 收集需求：必须使用Kaiwu Form MCP
- 技术调研、竞品分析：必须使用WebSearch MCP

## 解决方案设计规则
- 解决方案
  - 设计需求的解决方案前：必须使用SearchCodebase分析现有代码逻辑和表结构
  - 功能类型：必须明确区分该需求是属于“新增”或者“迭代”

## React+组件库开发规则（重要，融合20年+ UI/UX设计经验）

### 开发规则（必须严格遵守）
- React页面开发（全局！）
  - 开发React页面前：必须先了解、熟悉现有的代码框架
  - 页面开发时：优先使用项目真实代码，有疑问时必须询问用户
  - 功能点交互：React页面里，每个功能点点击时，必须得有交互和对应的交互页面
  - 页面数据：开发的React页面时，必须同步维护50条业务数据
  - **用户体验**：必须遵循UI/UX设计原则，注重易用性和满意度
  - **视觉设计**：必须遵循设计系统，保持视觉一致性
  - **交互细节**：必须关注每一个交互细节，提供良好的操作反馈

1. **"React+Ant Design"方式**
- 开发页面时
  - 必须使用React+Ant Design组件库开发页面
  - 必须严格遵守Ant Design设计规范
  - **必须遵循UI/UX设计原则**：
    - 用户研究：理解目标用户特征、使用场景、痛点需求
    - 信息架构：清晰的信息层级、导航设计、搜索设计
    - 交互设计：操作反馈、防错设计、效率优化
    - 视觉设计：色彩系统、字体系统、间距系统、视觉层次
  - **必须关注设计细节**：
    - 按钮规范：主按钮、次按钮、文字按钮的合理使用
    - 表单规范：输入框、选择器、日期选择的统一风格
    - 表格规范：列宽、排序、筛选、分页的合理布局
    - 响应式设计：断点适配、布局调整、触控优化
- 开发的React页面截图放入PRD文档
- **设计说明**：必须提供UI设计说明，包括设计思路、关键决策、用户体验优化点
- **字段设计规范**：页面字段必须与业务流程关联，参见 `pm_skills/SKILL.md` 技能5中的"字段设计规范"

## UI/UX设计质量检查规则（重要）

### 设计审查清单（必须逐项检查）
- **用户体验检查**
  - 是否从用户视角出发设计？
  - 操作流程是否简洁明了？
  - 是否有清晰的错误提示和反馈？
  - 是否考虑了异常场景和边界情况？
  - 是否符合Nielsen十大可用性原则？

- **视觉设计检查**
  - 色彩使用是否符合设计系统？
  - 字体大小、字重是否层次清晰？
  - 间距是否统一（8px网格系统）？
  - 视觉层次是否明确（重要信息突出）？
  - 是否有足够的留白（呼吸感）？

- **交互设计检查**
  - 每个操作是否有明确的反馈？
  - 按钮状态（默认、悬浮、点击、禁用）是否完整？
  - 表单校验是否即时、明确？
  - 加载状态、空状态、错误状态是否设计？
  - 是否有防错设计（确认对话框、撤销操作）？

- **响应式设计检查**
  - 是否适配不同屏幕尺寸？
  - 移动端触控区域是否足够大（最小44px）？
  - 表格在移动端是否有横向滚动或卡片化？
  - 侧边栏在移动端是否可折叠？

### 设计交付标准
- 必须提供设计说明文档
- 必须说明设计思路和决策依据
- 必须列出用户体验优化点
- 必须提供交互流程图（复杂功能）

## 需求-方案匹配验证循环规则（自动闭环机制）

### 循环位置
本验证循环发生在**阶段二：细化方案**内部，确保方案与需求高度匹配后再进入阶段三执行。

### 循环流程
```
需求输入 → 需求分析 → 方案设计 → 匹配验证 → 是否匹配？
                                              ↓
                                    是 → 输出最终方案
                                    否 → 分析偏差 → 调整方案 → 再次验证
```

### 阶段定义

#### 阶段1：需求分析（Analysis）
- 提取需求核心要点清单
- 识别功能边界和约束条件
- 明确验收标准和成功指标
- 输出：《需求要点清单》

#### 阶段2：方案设计（Design）
- 基于需求设计解决方案
- **输出内容包括**：
  - React 开发的页面（UI 设计 + 交互实现）
  - 功能清单（详细功能点列表）
  - 业务流程图（Mermaid 流程图）
  - PRD 需求文档（完整需求规格说明）
- 输出：《方案设计文档》

#### 阶段3：匹配验证（Validation）
**验证条件检查项**：
- ✅ 功能覆盖度：方案功能是否覆盖所有需求功能点
- ✅ 业务流程：方案流程是否符合需求描述
- ✅ 边界条件：异常情况、边界场景是否处理
- ✅ 验收标准：方案是否满足需求的成功指标
- ✅ 约束条件：技术约束、业务约束是否满足
- ✅ 用户体验：是否符合用户需求和期望

**匹配度评分**：
- 100%：完全匹配，无需调整
- 90%-99%：轻微偏差，可选调整
- 70%-89%：明显偏差，必须调整
- <70%：严重偏差，重新设计

#### 阶段4：偏差分析（Gap Analysis）
- 识别未覆盖的功能点
- 分析偏离需求的设计
- 记录遗漏的边界条件
- 输出：《偏差分析报告》

#### 阶段5：方案调整（Adjustment）
- 针对偏差进行设计调整
- 补充缺失的功能点
- 修正偏离的业务流程
- 输出：《调整后的方案》

### 循环控制规则

#### 终止条件（满足任一即终止）
1. **匹配度 ≥ 95%**：方案与需求高度匹配
2. **迭代次数 ≥ 5次**：防止无限循环
3. **连续2次无改进**：方案已无法优化
4. **用户人工确认**：用户认可当前方案

#### 最大迭代次数
- 默认：5次
- 复杂需求：可延长至8次
- 简单需求：可缩短至3次

#### 偏差分级处理
| 级别 | 偏差程度 | 处理方式 | 示例 |
|------|---------|---------|------|
| P0 | 核心功能缺失 | 必须调整 | 需求要求搜索功能，方案未设计 |
| P1 | 功能描述不清 | 澄清后调整 | 需求说"快速查询"，方案需明确具体时效 |
| P2 | 优化建议 | 可选调整 | 建议增加导出功能，但需求未明确要求 |

### 输出要求

#### 每次循环必须输出
1. **当前迭代次数**：第 X 次验证
2. **匹配度百分比**：XX%
3. **偏差功能点清单**：
   - 未覆盖的功能点
   - 偏离需求的设计
   - 遗漏的边界条件
4. **调整建议**：针对偏差的改进方案
5. **下次验证重点**：需要特别关注的检查项

#### 最终输出
- 《需求-方案匹配验证报告》
- 《最终方案设计文档》
- 《偏差处理记录》（如有）

### 执行规则

1. **自动触发**：需求设计完成后自动进入验证循环
2. **人工介入**：达到终止条件时询问用户确认
3. **记录存档**：每次循环结果保存到项目文档
4. **复盘总结**：循环结束后总结优化经验

## 不包含技术性设计内容
### 产品经理不应该设计的内容
- ❌ 后端接口设计（API接口定义、请求参数、响应格式）
- ❌ 数据库表结构设计（表名、字段名、字段类型、索引、关联关系）
- ❌ 具体技术实现方案（前端框架、后端框架、中间件）
- ❌ 技术选型（编程语言、数据库、缓存、消息队列）
- ❌ 性能优化方案（SQL优化、缓存策略、并发处理）
- ❌ 安全设计方案（权限控制、数据加密、防攻击）
- ❌ 部署架构设计（服务器配置、容器化、CI/CD）
